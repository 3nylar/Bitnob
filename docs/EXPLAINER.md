# AMM Liquidity Pool Dashboard — Frontend Explainer

This document outlines the architecture, mathematical implementations, and data structures used in the React/TypeScript frontend implementation of the Constant Product Automated Market Maker (AMM) Simulation Dashboard.

---

## 1. Core Engineering Principles

The frontend acts as a real-time reactive interface for a localized Python AMM engine running a constant product pool formula ($x \cdot y = k$).

* **Stateless Operations & Consensus:** Execution state belongs strictly to the backend engine. The interface manages optimistic user interface inputs, handles real-time data visualizer streaming via Recharts, and reads contextual session hashes (`user_id`) to sync asset positions.
* **Asynchronous Serialization:** Race conditions on simultaneous user interaction updates and periodic intervals are structurally isolated using a tracking sequence ref (`fetchSeqRef`), dropping outdated asynchronous callbacks gracefully.

## 3. Client Calculations & Mathematical Formulations

To preserve reactive fluid workflows without making redundant API over-fetches, standard constant product formulas are evaluated in-memory using React `useMemo` hooks before execution confirmation.

### A. Swap Quotation Engine

Trades invoke a standard constant product formula modifying inputs to account for the platform's **30 basis point (0.30%) fee**:

$$\text{amountInAfterFee} = \Delta x \cdot \left(1 - \frac{30}{10000}\right)$$

The resultant token output ($\Delta y$) generated by the state shift is estimated using:

$$\Delta y = \frac{\text{amountInAfterFee} \cdot y}{x + \text{amountInAfterFee}}$$

```typescript
function quoteOutput(amountIn: number, reserveIn: number, reserveOut: number): number {
  if (!amountIn || amountIn <= 0 || reserveIn <= 0 || reserveOut <= 0) return 0;
  const amountInAfterFee = amountIn * (1 - 30 / 10000); // 30 BPS
  return (amountInAfterFee * reserveOut) / (reserveIn + amountInAfterFee);
}
```

### B. Proportional Liquidity Rules

Liquidity deposits must be strictly symmetric to prevent initiating artificial arbitrage price shocks. The frontend enforces a static ratio requirement:

$$\Delta y_{\text{required}} = \Delta x \cdot \left(\frac{\text{reserve}_b}{\text{reserve}_a}\right)$$

LP shares minted to track pool ownership follow a standard relative weight distribution mechanism:

$$\text{Shares Minted} = \text{Total Shares} \cdot \left(\frac{\Delta x}{\text{reserve}_a}\right)$$

## 4. Operational Feature Architecture

### 1. Persistent User Tracking

User profiles rely on stateless execution tracking. The platform calls `getOrCreateUserId()` to evaluate `window.localStorage` for a cached random identification string. If unavailable, a `crypto.randomUUID()` fallback string is instantiated to decouple multi-account activity.

### 2. Guardrails Against Slippage and Volatility

* **Price Impact Mitigation:** If a user keys in a swap value exceeding **50%** of total system pool reserves, an explicit alert boundary warning (`insufficientLiquidity`) warns of heavy price degradation.
* **Slippage Invariant Validation:** Following confirmation endpoints, the code validates actual backend token distribution weights against selected user tolerances (`slippagePct`). If unexpected execution parameters cause output amounts below the computed threshold, detailed client alerts are triggered.

### 3. Price History Streams

The interface charts macro performance by tracking `pool.price_a` on every subsequent synchronization block. The chart maintains a maximum buffer window history size array up to **200 evaluation ticks** before clean garbage collection array slicing (`MAX_PRICE_HISTORY`) takes effect.
